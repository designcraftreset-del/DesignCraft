# Соответствие таблиц БД спецификации (раздел 6.1)

Ниже перечислены отличия реальной структуры базы DesignCraft от описанной в спецификации. Что совпадает — не перечислено.

---

## 1. Таблица «Пользователи» (users)

| В спецификации | В проекте | Комментарий |
|----------------|-----------|-------------|
| — | `email_verified_at` | Есть в миграции (Laravel по умолчанию). |
| — | `last_seen_at` | Добавлено в проекте (последний визит). |

Остальное совпадает: id, name, email, password, role, avatar, phone, created_at, updated_at, remember_token.

---

## 2. Таблица «Услуги» (services)

Структура в проекте другая.

| В спецификации | В проекте | Комментарий |
|----------------|-----------|-------------|
| name | title, titleTwo | Название разбито на два поля. |
| description | subtitle, formatTwo, term | Описание в нескольких полях. |
| type | category | Тип услуги хранится как category. |
| base_price | money | Стоимость в поле money (строка). |
| image | — | Нет поля изображения-превью. |
| is_active | — | Нет поля активности. |
| — | concept, edits | Есть в проекте. |

Для соответствия спецификации можно добавить колонки: `image` (путь к превью), `is_active` (boolean), при необходимости переименовать/объединить поля.

---

## 3. Таблица «Пакеты услуг» (packages)

В проекте таблицы `packages` нет.

- В заказах используется строка: в таблице `applications` есть колонка `paket` (название пакета), без связи с отдельной таблицей пакетов.
- Для полного соответствия спецификации нужно: создать таблицу `packages` (id, service_id, name, price, description, delivery_time, timestamps) и в заказах хранить `package_id` вместо/вместе с `paket`.

---

## 4. Таблица «Заказы» (orders)

В проекте заказы хранятся в таблице **`applications`**, не `orders`. Имена полей и набор колонок отличаются.

| В спецификации (orders) | В проекте (applications) | Комментарий |
|-------------------------|---------------------------|-------------|
| Таблица orders | Таблица **applications** | Другое имя таблицы. |
| user_id | userid | Другое имя (без подчёркивания). |
| service_id | — | Нет, тип услуги в строке `yslyga`. |
| package_id | paket | Строка (название пакета), не внешний ключ. |
| description | info | Описание задачи. |
| source_files | sources_path | Путь к исходникам. |
| result_files | — | Нет отдельного поля (есть preview_path, sources_path, photos_paths). |
| moderator_id | — | Нет закреплённого модератора. |
| deadline | — | Нет срока сдачи. |
| — | name, email, nomer | Дублирование контактов заказчика. |
| — | preview_path, photos_paths, chat_closed_at, preview_sent_at | Специфичные для проекта поля. |

Для соответствия спецификации: либо переименовать таблицу/поля и добавить service_id, package_id, moderator_id, deadline, result_files; либо оставить applications и в документе указать, что «заказы» реализованы как таблица applications с перечисленными полями.

---

## 5. Таблица «Портфолио» (portfolios)

В проекте таблицы `portfolios` нет.

- Работы для портфолио хранятся в таблице **`banners`** (id, title, subtitle, description, image_path, category, user_id, is_approved, timestamps). По смыслу это и есть «портфолио» в текущей реализации.
- Для соответствия спецификации можно: создать таблицу `portfolios` по описанию из 6.1 и перенести/продублировать данные из banners, либо в документе описать, что портфолио реализовано через таблицу banners с перечислением полей.

| В спецификации | В проекте | Комментарий |
|----------------|-----------|-------------|
| preview | excerpt | Краткое превью. |

Остальное совпадает (id, title, slug, content, created_at, updated_at). Имена можно считать эквивалентными (preview ↔ excerpt, image ↔ image_path, views ↔ views_count).

## 9. Таблица «Чаты» (chats)

В проекте отдельной таблицы `chats` с полями type, order_id, user_id, moderator_id, is_active нет.

- Реализовано иначе:
  - **admin_chat_messages** — общий админ-чат (без привязки к заказу в самой таблице).
  - **support_threads** — чаты поддержки (по одному на пользователя, связь с user_id).
  - Чат по заказу хранится в контексте заявки (applications: chat_closed_at и т.д.), без отдельной таблицы chats с order_id.

Для соответствия спецификации нужна таблица `chats` (id, type, order_id, user_id, moderator_id, is_active, timestamps) и при необходимости перенос/связь с текущими сообщениями.

---

## 10. Таблица «Сообщения» (messages)

В проекте нет единой таблицы `messages` с полем chat_id.

- Есть **admin_chat_messages** (id, user_id, message, is_system, read_at, chat_type, thread_id, image_path, image_name, file_path, file_name, timestamps).
- Сообщения поддержки хранятся в своей структуре (support_threads и связанные таблицы).
- Сообщения по заказам — в своей логике (order chat).

Для соответствия спецификации нужна таблица `messages` (id, chat_id, user_id, message, file_path, is_read, timestamps) и привязка к таблице `chats` из п. 9.

---

## Краткая сводка

| Спецификация | В проекте | Действие для соответствия |
|--------------|-----------|---------------------------|
| users | users | Почти совпадает (есть email_verified_at, last_seen_at). |
| services | services | Другая структура; нет image, is_active; другие имена полей. |
| packages | нет | Есть только paket в applications. Нужна таблица packages. |
| orders | **applications** | Другое имя таблицы и полей; нет service_id, package_id, moderator_id, deadline, result_files. |
| portfolios | нет (есть banners) | Портфолио = banners; при необходимости ввести таблицу portfolios. |
| news | news | Совпадает по смыслу (preview→excerpt, image→image_path, views→views_count); в проекте есть доп. поля. |
| reviews | reviews | Нет order_id; comment→review_text; есть client_name, client_position. |
| banners | banners | Нет link, position, is_active; есть subtitle, description, category, user_id, is_approved. |
| chats | нет | Чаты реализованы через admin_chat_messages, support_threads и логику заказов. Нужна таблица chats при следовании спецификации. |
| messages | нет (есть admin_chat_messages и др.) | Нет единой messages с chat_id. Нужна при следовании спецификации. |

Если нужно, можно подготовить миграции и правки моделей под полное соответствие спецификации (новые таблицы packages, portfolios, chats, messages и переименование/добавление полей в applications, services, banners, reviews).
