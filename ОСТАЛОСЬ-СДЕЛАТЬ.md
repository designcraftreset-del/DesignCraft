# Обновление сайта на Render и импорт базы

**Сайт:** https://designcraft-xej3.onrender.com

---

## Как выкатить изменения на сайт (чтобы сайт поменялся)

1. Откройте **терминал** (PowerShell или cmd): в Cursor/VS Code — меню Terminal → New Terminal, либо Win+R → `cmd` → Enter.

2. Перейдите в папку проекта и выполните команды по очереди:

```
cd c:\ospanel\domains\DesignCraft
git add .
git commit -m "Обновление сайта"
git push origin main
```

   - Если папка уже `c:\ospanel\domains\DesignCraft`, первую строку можно не вводить.
   - В `commit -m "..."` можно написать своё описание изменений.
   - Если попросит логин/пароль Git — введите данные от GitHub (или от того репозитория, куда пушите).

3. Зайдите на https://dashboard.render.com → ваш сервис DesignCraft. Должен быть включён **Auto-Deploy** (при push в ветку `main` деплой запускается сам).

4. Подождите 1–2 минуты. В Render во вкладке **Events** появится деплой и статус «Live».

5. Откройте в браузере https://designcraft-xej3.onrender.com — сайт уже с новым кодом (главная, /register, админка и т.д.).

## Актуальные исправления (уже в коде)

- **Админка → Сообщения** — мигание списка убрано: опрос раз в 4 с только добавляет новые сообщения (без перерисовки всего списка). При ошибке отправки ответа показывается уведомление.
- **500 в дашборде админки** — для PostgreSQL используются запросы с EXTRACT (месяц/день/год) вместо MONTH/YEAR/DAY.
- **500 в чате поддержки** — безопасная работа с датами сообщений и last_seen.
- **500 после регистрации** — отключена отправка верификации; регистрация с обработкой ошибок и редиректом.
- **Регистрация — понятные ошибки** — при занятом email показывается «Этот email уже зарегистрирован. Войдите или восстановите пароль.»; остальные ошибки БД логируются (Render → Logs).
- **Админка → Товары/услуги** (`/adminPanel2?page=products`) — добавлена кнопка **Удалить** у каждой услуги с подтверждением; маршрут `DELETE /services/{id}`.
- **Фото на бесплатном хосте не пропадают** — новости, аватары, баннеры и фото заказов сохраняются в **папку сайта** `public/uploads/` (news, avatars, banners, order_photos). Эти файлы попадают в репозиторий и не стираются при «засыпании» Render. Если файла нет (например, ещё не добавлен в репо), показывается заглушка.

## Фото и файлы в папке сайта (чтобы не пропадали на бесплатном хосте)

На бесплатном тарифе Render при «засыпании» сервиса стирается содержимое `storage/`. Поэтому загрузки перенесены в **`public/uploads/`** — эта папка входит в репозиторий и при деплое подтягивается на сервер.

- **Новости** — картинки в `public/uploads/news/`
- **Аватары** — в `public/uploads/avatars/`
- **Баннеры (портфолио)** — в `public/uploads/banners/`
- **Фото к заказу** — в `public/uploads/order_photos/`

**Как добавить фото, чтобы они не пропадали:** положите файлы в нужную подпапку `public/uploads/...`, закоммитьте и запушьте (см. раздел «Как выкатить изменения на сайт»). После деплоя они будут отображаться на сайте. Новые загрузки через форму на сайте тоже пишутся в `public/uploads/`, но на Render при следующем деплое или перезапуске файлы с диска стираются — чтобы сохранить их надолго, скачайте добавленные файлы и положите в репозиторий в `public/uploads/`, затем снова сделайте commit и push.

## Импорт базы (на всякий случай)

Маршрут **`/import-db`** оставлен в коде. Если понадобится снова залить данные из дампа:

1. Локально: обновите JSON в `database/db-export` (например `php artisan db:export`, затем скопируйте файлы из `storage/app/db-export` в `database/db-export`).
2. Закоммитьте и запушьте:
   ```bash
   git add database/db-export/
   git commit -m "Update db export"
   git push origin main
   ```
3. В **Environment** на Render добавьте переменную `IMPORT_DB_TOKEN` = любой секрет.
4. После деплоя откройте один раз: `https://designcraft-xej3.onrender.com/import-db?token=ВАШ_ТОКЕН`
5. После импорта удалите `IMPORT_DB_TOKEN` из Environment.
6. Если не входится под админом — добавьте `SETUP_ADMIN_TOKEN`, откройте `/setup-admin?token=...` один раз, затем удалите переменную.

## Чтобы письма отправлялись (сброс пароля и др.) на Render

Локально в `.env` уже настроен Gmail SMTP — письма уходят. На Render переменные из `.env` не подставляются (файл не в репозитории), поэтому почту нужно задать в **Environment** в панели Render.

1. Зайдите в **Render Dashboard** → ваш сервис DesignCraft → **Environment**.
2. Добавьте переменные (значения возьмите из вашего локального `.env`):

| Переменная | Пример значения |
|------------|-----------------|
| `MAIL_MAILER` | `smtp` |
| `MAIL_HOST` | `smtp.gmail.com` |
| `MAIL_PORT` | `587` |
| `MAIL_USERNAME` | ваш Gmail (например `designcraftreset@gmail.com`) |
| `MAIL_PASSWORD` | **пароль приложения** Gmail (не обычный пароль) |
| `MAIL_ENCRYPTION` | `tls` |
| `MAIL_FROM_ADDRESS` | тот же email, что и MAIL_USERNAME |
| `MAIL_FROM_NAME` | `DesignCraft` |

3. **Пароль приложения Gmail:** Google → [Аккаунт](https://myaccount.google.com/) → Безопасность → Двухэтапная аутентификация (включить) → Пароли приложений → создать пароль для «Почта». Этот пароль вставьте в `MAIL_PASSWORD` на Render.
4. Сохраните Environment и дождитесь перезапуска сервиса (или нажмите **Manual Deploy**).

После этого ссылка «Забыли пароль» будет отправлять письмо на указанный пользователем email. Если письмо не уходит — в Render → **Logs** ищите строки `Password reset link send failed:` — там будет причина (например, блокировка Gmail, неверный пароль приложения).

## Ошибка «Tablespace is missing for table applications» (MySQL 1812)

Эта ошибка значит, что у таблицы `applications` повреждено или отсутствует табличное пространство InnoDB (файл .ibd). **Сайт при этом больше не падает** — загрузка заявок подменяется пустым списком, главная открывается.

**Как восстановить таблицу:**

1. **Есть бэкап БД** — восстановите из него базу или только таблицу `applications`.
2. **Нет бэкапа** — пересоздайте таблицу по миграциям:
   - В phpMyAdmin или консоли MySQL выполните `SHOW CREATE TABLE applications;` (если таблица ещё открывается) и сохраните структуру.
   - Или выполните миграции заново: `php artisan migrate:fresh` (удалит все данные) или создайте таблицу вручную по существующей миграции в `database/migrations/`.
3. **Локально (OSPanel)** — проверьте, что каталог данных MySQL не удалялся и сервер MySQL запущен. При необходимости пересоздайте базу и импортируйте дамп.

После восстановления таблицы заявки снова будут подгружаться на главной и в личном кабинете.

## Если 500 после регистрации или в админ-панели

В коде исправлен middleware обновления «последний визит» (UpdateUserLastSeen). Убедитесь, что на Render задеплоена последняя версия (`git push origin main`), затем снова зарегистрируйтесь или откройте админку. Если 500 остаётся — откройте в Render **Logs** и посмотрите текст ошибки в момент действия.

**Сообщение «Ошибка при регистрации»** — пользователь видит его при сбое БД или при занятом email. После обновления кода при занятом email показывается «Этот email уже зарегистрирован...». Остальные причины смотрите в Logs по строкам `Registration DB error` / `Registration failed`.
